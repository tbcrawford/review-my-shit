<decisions>
## Implementation Decisions

### Architecture

- **D-01:** Hybrid Node.js architecture. Slash command files in `.opencode/commands/` and `.cursor/commands/` are thin entry points that call into a Node.js orchestration script via `!`node ...``. The Node.js layer uses the Vercel AI SDK to make three isolated `generateText` calls — one per agent role (reviewer, validator, writer).
- **D-02:** Separate command per scope. There are two commands: `review-local` (staged/unstaged git diff) and `review-pr` (GitHub PR diff). Each is a separate `.md` file in both editors' commands directories.
- **D-03:** Agent prompts are inline in the Node.js orchestrator. No AGENTS.md, no separate markdown agent files. The orchestrator script owns the prompt for each of the three roles.

### Installation & Distribution

- **D-04:** Distributed as an npm package, designed for global install via `npx rms@latest`. Running `npx rms@latest` again updates the global install (idempotent).
- **D-05:** The npm package binary is the installer. When run globally (`npx rms@latest --global`), it installs the binary. When run in a project, it writes editor command files into `.opencode/commands/` and `.cursor/commands/` — mirroring the GSD install pattern (`https://github.com/gsd-build/get-shit-done`).
- **D-06:** `rms install` (or `npx rms install`) in a project writes command files for both OpenCode and Cursor in a single step. Both editors are supported from day one.

### Finding ID Scheme

- **D-07:** Dimension-prefixed counter format. Finding IDs are formed as `{DIM}-{NNNNN}` where:
  - `{DIM}` is a 3-4 character abbreviation for the review dimension (e.g. `SEC`, `BUG`, `PERF`, `ARCH`, `ERR`, `DATA`, `API`, `DEP`, `DOC`, `TEST`, `STYL`)
  - `{NNNNN}` is a zero-padded global counter (5 digits, e.g. `00001`)
  - Example: `SEC-00001`, `BUG-00002`, `PERF-00003`
- **D-08:** Global counter, never resets. The counter is stored in `.reviews/.counter` (or equivalent) and increments across all sessions. Every finding ever generated gets a unique ID. This supports the fix command's by-ID mode and prevents audit trail collisions across review sessions.
- **D-09:** Finding IDs are generated by the Node.js orchestrator (deterministic), not by the LLM agents. The reviewer LLM does not generate IDs — the orchestrator assigns them after parsing structured output.

### Schema File Structure

- **D-10:** All pipeline files (INPUT, REVIEWER, VALIDATOR, REPORT) use XML-tagged blocks for structured data. This provides unambiguous delimiters for the Node.js parser and limits prompt injection surface (attacker code cannot break out of a tagged section).
- **D-11:** INPUT file uses XML-tagged sections: `<diff>`, `<scope>`, `<focus>`. Consistent parse pattern throughout the pipeline.
- **D-12:** REPORT.md also uses XML tags. The `/fix` command can parse REPORT.md directly without reading intermediate files (REVIEWER.md, VALIDATOR.md). REPORT.md is the single source of truth for fix operations.
- **D-13:** Human-readable markdown prose exists alongside the XML structure in REPORT.md (severity group headings, finding summaries rendered as readable text). The XML tags carry the machine-readable structured data; prose provides the human reading layer.

### the Agent's Discretion

- Exact dimension abbreviation list (11 abbreviations for the 11 review dimensions) — follow established conventions where they exist (SEC, PERF, etc.), pick readable 3-4 char abbreviations for the rest.
- Counter file location within `.reviews/` (e.g. `.counter`, `counter.json`, or similar).
- Whether the npm package name is `rms` or `review-my-shit` — pick whichever is available on npm.
- Command file frontmatter details (description wording, argument-hint text).

</decisions>

<objective>
Add an interactive TUI settings picker for `rms settings` (invoked with no flags) and update
both editor command templates to document the new copilot provider.

Purpose: Users who discover rms for the first time have no way to know the `provider:model`
format. A TUI picker with curated model lists removes the need to know internal IDs. Copilot
users especially benefit — they can pick "copilot" → "claude-opus-4.6" without knowing
the `opencode models` format exists.

Output:
- `src/settings-tui.ts` (NEW) — three-step `@inquirer/prompts` flow: agent → provider → model → confirm → save
- `src/index.ts` — wire TUI into the no-flags branch of the settings command
- `src/templates/opencode-settings.md` — update argument-hint and description
- `src/templates/cursor-rms-settings/SKILL.md` — copilot provider docs + Cursor model ID dead-end warning
</objective>

    ## Supported providers

    | Provider | Description | Requires |
    |----------|-------------|---------|
    | `copilot` | GitHub Copilot (OpenAI-compatible) | `GITHUB_TOKEN` or opencode auth |
    | `anthropic` | Direct Anthropic API | `ANTHROPIC_API_KEY` |
    | `openai` | Direct OpenAI API | `OPENAI_API_KEY` |
    | `google` | Direct Google AI API | `GOOGLE_GENERATIVE_AI_API_KEY` |

    ## Cursor model IDs cannot be used directly

    `cursor-agent --list-models` outputs Cursor-internal routing IDs (e.g. `claude-4.6-sonnet-medium`)
    that only work inside Cursor's proprietary backend. There is no public API endpoint for these IDs.

    If you have GitHub Copilot, use the `copilot` provider instead:
    ```
    rms settings --reviewer copilot:claude-opus-4.6
    ```

    Otherwise, use a provider with a direct API key (anthropic, openai, or google).

    ## Interactive picker

    Run `rms settings` with no arguments in a terminal for a guided interactive picker.
    The picker requires a TTY — it falls back to help text in non-TTY environments.

    The config is saved to ~/.config/rms/config.json. If no config exists, rms falls back
    to AI_SDK_PROVIDER + AI_SDK_MODEL env vars.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| TTY stdin → select() | User arrow-key input flows into @inquirer/prompts; choices are from a static whitelist |
| TUI output → saveRmsConfig() | Selected values are from a closed static MODELS list before being written to disk |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-05 | Tampering | runSettingsTui MODELS list | accept | Model list is static in code, not loaded from network. User cannot inject arbitrary model IDs through the TUI (only through --flags, which is existing behavior). |
| T-16-06 | Denial of Service | ExitPromptError in non-TTY | mitigate | Caught explicitly — falls through to help text rather than crashing with unhandled rejection. |
| T-16-07 | Information Disclosure | help text in non-TTY | accept | Help text reveals provider names and example model IDs — all public information. No secrets in help text. |
</threat_model>

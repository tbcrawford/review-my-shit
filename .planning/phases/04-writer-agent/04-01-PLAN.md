# Plan 04-01: Writer Core — parseCounterFindings + runWriter

**Phase:** 04-writer-agent
**Plan:** 04-01
**Status:** pending

## Goal

Implement `parseCounterFindings` in `pipeline-io.ts` and the full `runWriter` function in a new `src/writer.ts`. This is the deterministic report assembly step: no LLM call, pure TypeScript template rendering from structured data.

## Requirements Covered

- PIPE-05: Writer agent synthesizes reviewer + validator output into final report
- REPT-01: Metadata header (scope, focus, model, timestamp, dimensions)
- REPT-02: Findings grouped by severity (critical → high → medium → low → info)
- REPT-03: Each finding has file path, line, explanation, suggestion, finding ID
- REPT-04: `.reviews/<session>/` contains REVIEWER.md, VALIDATOR.md, REPORT.md

## Success Criteria

This plan is done when:
1. `parseCounterFindings(rawContent)` correctly extracts `<counter-finding>` blocks from VALIDATOR.md rawContent
2. `runWriter` produces REPORT.md with findings grouped by severity
3. Every finding entry contains: ID, file, line, dimension, explanation, suggestion, source attribution
4. Challenged findings appear in report with validator rationale label
5. Escalated findings have severity bumped one level with annotation
6. Counter-findings appear as separate entries attributed to validator with new IDs
7. Metadata header matches REPT-01 fields
8. TypeScript compiles clean (`npm run build`)

## Tasks

### Task 1: Add `parseCounterFindings` to `pipeline-io.ts`

Add a new exported function after `parseValidatorOutput`:

```typescript
export function parseCounterFindings(rawContent: string): Array<Omit<Finding, 'id'>>
```

- Use regex `/<verdict>([\s\S]*?)<\/verdict>/g` to get verdict blocks
- For each challenged verdict block, extract nested `<counter-finding>...</counter-finding>`
- Pass counter-finding content through `parseFindingBlock` (reuse existing internal function — make it exported or duplicate the logic)
- Validate each counter-finding with `FindingSchema.omit({ id: true }).safeParse()`
- Skip invalid counter-findings with `console.warn`
- Return array of valid counter-findings (without id — IDs assigned by runWriter)

### Task 2: Create `src/writer.ts`

New file. Structure mirrors `src/validator.ts` (options interface, result interface, main function).

**Interfaces:**

```typescript
export interface WriterOptions {
  session: SessionInfo;
  findings: Finding[];                          // from runReviewer (with IDs)
  verdicts: ValidationVerdict[];                // from runValidator
  validatorRawContent: string;                  // for parseCounterFindings
  inputMdContent: string;                       // for scope/focus/timestamp
  dimensionsCovered: Dimension[];               // from parseReviewerOutput
  modelId: string;                              // passed from index.ts
}

export interface WriterResult {
  reportMdPath: string;
  findingCount: number;       // total findings in report (reviewer + counter)
  counterFindingCount: number;
}
```

**`runWriter` logic:**

1. Parse counter-findings from `validatorRawContent` via `parseCounterFindings`
2. Assign IDs to counter-findings using the existing ID assignment module
3. Build the merged finding list:
   - Start with all reviewer `findings`
   - Apply escalation: for each `escalated` verdict, bump severity one level on the corresponding finding
   - Attach verdict annotations to each finding (challenged label + rationale, escalated label + rationale)
   - Append counter-findings (attributed to validator)
4. Sort findings: group by severity (`critical → high → medium → low → info`), then by dimension, then by file
5. Parse INPUT.md frontmatter to extract scope, focus, timestamp
6. Render REPORT.md using a template string:
   - YAML frontmatter (matches `ReportFileSchema`)
   - Metadata header section (human-readable)
   - Severity-grouped finding sections
7. Validate frontmatter against `ReportFileSchema.safeParse()` before writing — throw if invalid
8. Write REPORT.md to `session.sessionDir`
9. Run completeness check: read file back, regex for all `[{ID}]` patterns, assert all reviewer finding IDs present — throw with missing IDs if any are absent
10. Return `WriterResult`

**Severity bump lookup:**
```typescript
const SEVERITY_BUMP: Record<Severity, Severity> = {
  info: 'low',
  low: 'medium',
  medium: 'high',
  high: 'critical',
  critical: 'critical',
};
```

**Severity sort order:**
```typescript
const SEVERITY_ORDER: Severity[] = ['critical', 'high', 'medium', 'low', 'info'];
```

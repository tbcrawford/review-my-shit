# rms — review-my-shit

A three-agent AI code review pipeline for OpenCode and Cursor. The reviewer catches problems a developer would miss. The validator catches problems the reviewer would miss. Both are fully auditable.

---

## How it works

Every review runs three agents in strict sequence, each in a fully isolated context window:

```
/rms-review [local | pr <number>]
        │
        ▼
  ┌─────────────┐
  │  Reviewer   │  Analyzes the diff across 11 dimensions. Writes REVIEWER.md.
  └──────┬──────┘
         │
         ▼
  ┌─────────────┐
  │  Validator  │  Adversarially challenges each finding. Writes VALIDATOR.md.
  └──────┬──────┘
         │
         ▼
  ┌─────────────┐
  │   Writer    │  Merges findings + verdicts into a severity-grouped REPORT.md.
  └──────┬──────┘
         │
         ▼
  .reviews/<session>/
    INPUT.md      ← scope, focus area, and diff
    REVIEWER.md   ← primary findings
    VALIDATOR.md  ← validator verdicts
    REPORT.md     ← final report
```

All three agents run as isolated `generateText` calls — no shared history, no context bleed between roles.

## Getting started

### Prerequisites

- Node.js 18+
- OpenCode or Cursor
- API key for your chosen AI provider (OpenAI, Anthropic, or Google)

### Install

```bash
npx review-my-shit
```

Running `npx review-my-shit` prompts you to choose which editors to install for (OpenCode, Cursor, or both), then writes the slash command files:

- **OpenCode:** `~/.config/opencode/command/` — global, available in all projects
- **Cursor:** `~/.cursor/skills/<name>/SKILL.md` — global, available in all projects

Re-run after upgrading to pick up updated commands.

**Scripting / CI:** Pass flags to skip the interactive prompt:

```bash
npx review-my-shit --opencode   # OpenCode only
npx review-my-shit --cursor     # Cursor only
npx review-my-shit --yes        # Both editors (non-interactive)
```

> **Already have rms installed globally?** You can also run `rms install` directly, with the same flags: `rms install --opencode`, `rms install --cursor`, `rms install -y`.

### Configure models

By default rms falls back to the `AI_SDK_PROVIDER` and `AI_SDK_MODEL` environment variables (all three agents share the same model). For per-agent control, use `/rms-settings`:

```
/rms-settings --reviewer anthropic:claude-opus-4-5
/rms-settings --validator anthropic:claude-sonnet-4-5
/rms-settings --writer openai:gpt-4o
```

This writes `~/.config/rms/config.json`, which takes precedence over env vars. View the current config by running `/rms-settings` with no arguments.

For PR reviews, set a GitHub Personal Access Token:

```bash
export GITHUB_TOKEN=ghp_...
```

## Commands

### `/rms-review [local | pr <number>] [--focus <area>]`

Unified review entry point. Prompts for scope if called with no arguments.

```
/rms-review               # prompts: local diff or PR?
/rms-review local         # review staged + unstaged git changes
/rms-review pr 42         # review GitHub PR #42
/rms-review local --focus security
```

### `/rms-fix [<finding-id>] [--session <id>]`

Display a finding and its suggestion so your AI agent can apply it.

```
/rms-fix                     # interactive — lists all findings from the latest report
/rms-fix SEC-00001           # jump directly to a specific finding
/rms-fix BUG-00003 --session 2026-04-06-local-a3b7
```

The fix command never applies changes automatically. It presents the finding and suggestion; you decide what to do.

### `/rms-settings [--reviewer p:m] [--validator p:m] [--writer p:m] [--reset]`

View or set per-agent model configuration.

```
/rms-settings                                  # show current config
/rms-settings --reviewer anthropic:claude-opus-4-5
/rms-settings --validator anthropic:claude-sonnet-4-5
/rms-settings --writer openai:gpt-4o
/rms-settings --reset                          # delete config, revert to env vars
```

Supported providers: `openai`, `anthropic`, `google`

## Review dimensions

The reviewer analyzes code across 11 dimensions:

1. Bugs & logic errors
2. Security
3. Performance
4. Style & conventions
5. Test coverage
6. Architecture
7. Error handling & resilience
8. Data integrity
9. API & interface contracts
10. Dependency & environment risk
11. Code & documentation consistency

Findings are grouped by severity in the final report: **Critical → High → Medium → Low → Info**.

Each finding includes a stable ID (e.g. `SEC-00001`), file path, line reference, explanation, and actionable suggestion.

---

## Finding IDs

IDs are assigned by the orchestrator after the LLM responds — never generated by the model. They are globally sequential and dimension-prefixed:

```
SEC-00001   BUG-00002   PERF-00003
```

The counter persists in `.reviews/.counter` across sessions. IDs are stable within a session.

## Output

```
.reviews/
├── .counter
└── 2026-04-06-local-a3b7/
    ├── INPUT.md
    ├── REVIEWER.md
    ├── VALIDATOR.md
    └── REPORT.md
```

`.reviews/` is automatically added to `.gitignore` on first run. Review artifacts are local only.

## CLI reference

```bash
# Unified review command (prompts for scope if none given)
rms review [local | pr <pr-number>] [--focus <area>]

# Show or apply a finding
rms fix [<finding-id>] [--session <id>]

# View or set per-agent model config
rms settings [--reviewer p:m] [--validator p:m] [--writer p:m] [--reset]

# Install slash commands
rms install
```

## Development

```bash
npm install       # install dependencies
npm run build     # compile TypeScript → dist/
npm test          # run tests (Node.js built-in test runner, no extra deps)
npm run typecheck # type-check without emitting
```

For development, use `node dist/index.js <command>` instead of the globally-installed `rms` to test local changes before publishing.

Tests cover: diff preprocessing, reviewer prompt structure, validator isolation, writer output, finding ID generation, pipeline I/O, session management, and config loading.
